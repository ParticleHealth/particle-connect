**labs_by_encounter.sql (CROSS-01) -- TEMPORAL JOIN:**

CRITICAL: Labs have NO encounter foreign key. Must use timestamp-based temporal join.

PostgreSQL version:
```
CTE parsed_encounters:
  - SELECT encounter_id, encounter_type_name,
    CAST("encounter_start_time" AS TIMESTAMPTZ) AS start_ts,
    CAST("encounter_end_time" AS TIMESTAMPTZ) AS end_ts,
    patient_id
  - FROM encounters
  - WHERE encounter_start_time IS NOT NULL AND encounter_end_time IS NOT NULL
    AND patient_id = '6f3bc061-...'

CTE parsed_labs:
  - SELECT lab_observation_id, lab_name, lab_value_quantity, lab_unit,
    CAST("lab_timestamp" AS TIMESTAMPTZ) AS lab_ts,
    patient_id
  - FROM labs
  - WHERE lab_timestamp IS NOT NULL
    AND patient_id = '6f3bc061-...'

Final SELECT:
  - INNER JOIN on patient_id AND lab_ts BETWEEN start_ts AND end_ts
  - Show encounter context alongside lab results
  - ORDER BY start_ts, lab_ts
```

BigQuery version: Same structure but:
- Backtick identifiers
- `PARSE_TIMESTAMP('%Y-%m-%dT%H:%M:%S%z', col)` for timestamps with +0000 offset
- SAFE_CAST for lab_value_quantity

**medications_by_problem.sql (CROSS-02) -- JOIN VIA ENCOUNTERS:**

The relationship path: encounters.condition_id_references (comma-separated) contains condition_id values from problems. Medications connect to practitioners, and encounters connect to both conditions and medications.

The most practical join: Use encounters as the bridge. Encounters reference conditions via condition_id_references. Medications are prescribed in the context of conditions. But there is no direct medication-to-condition FK.

Strategy: Join encounters (which have condition_id_references) to problems (via unnested condition_id_references matching condition_id), then show which medications were prescribed by practitioners involved in those encounters (via practitioner_role_id_references).

PostgreSQL version:
```
CTE encounter_conditions:
  - SELECT encounter_id, encounter_type_name,
    unnest(string_to_array("condition_id_references", ', ')) AS condition_id,
    patient_id
  - FROM encounters
  - WHERE condition_id_references IS NOT NULL
    AND patient_id = '6f3bc061-...'

CTE condition_details:
  - JOIN encounter_conditions to problems ON condition_id
  - SELECT encounter_id, condition_name, condition_clinical_status

CTE encounter_practitioners:
  - SELECT encounter_id,
    unnest(string_to_array("practitioner_role_id_references", ', ')) AS practitioner_role_id
  - FROM encounters
  - WHERE practitioner_role_id_references IS NOT NULL

CTE practitioner_medications:
  - JOIN medications ON practitioner_role_id to get medications prescribed by encounter practitioners
  - SELECT medication_name, medication_statement_status, practitioner_role_id

Final:
  - JOIN condition_details with practitioner_medications through encounter_id + encounter_practitioners
  - Show: condition_name, medication_name, encounter_type_name, medication_status
  - ORDER BY condition_name, medication_name
```

If the join through practitioners yields empty results (practitioners may not overlap), provide a simpler alternative as a commented second query: just show conditions and medications for the same patient side-by-side, noting the limitation that direct linkage is not available in flat data.

BigQuery version: Same structure but use SPLIT + UNNEST instead of string_to_array + unnest. Backtick identifiers.

Procedures have encounter_reference_id as a direct FK to encounters. However, all values are NULL in sample data. Still write the query correctly -- it will work when data has non-NULL encounter references.

PostgreSQL version:

```
CTE procedure_details:
  - SELECT procedure_id, procedure_name, procedure_code,
    CAST("procedure_date_time" AS TIMESTAMPTZ) AS procedure_ts,
    encounter_reference_id,
    performer_practitioner_role_reference_id,
    patient_id
  - FROM procedures
  - WHERE patient_id = '6f3bc061-...'

CTE encounter_details:
  - SELECT encounter_id, encounter_type_name,
    CAST("encounter_start_time" AS TIMESTAMPTZ) AS start_ts,
    CAST("encounter_end_time" AS TIMESTAMPTZ) AS end_ts
  - FROM encounters
  - WHERE patient_id = '6f3bc061-...'

CTE practitioner_details:
  - SELECT practitioner_role_id, practitioner_given_name, practitioner_family_name,
    practitioner_role_specialty
  - FROM practitioners
  - WHERE patient_id = '6f3bc061-...'

Final:
  - LEFT JOIN procedure_details to encounter_details ON encounter_reference_id = encounter_id
  - LEFT JOIN to practitioner_details ON performer_practitioner_role_reference_id = practitioner_role_id
  - Show: procedure_name, procedure_ts, encounter_type_name (may be NULL), practitioner name, specialty
  - Add comment: "Note: encounter_reference_id is NULL for all procedures in sample data.
    When populated, this join will link procedures to their encounters."
  - ORDER BY procedure_ts DESC NULLS LAST
```

Also add a commented-out alternative temporal join (similar to labs_by_encounter) for cases where encounter_reference_id is NULL but timestamps overlap.

BigQuery version: Same structure with backticks, PARSE_TIMESTAMP, SAFE_CAST.

```markdown
# Analytics Query Library

Ready-to-run SQL queries for Particle flat data. Each query is available in both PostgreSQL and BigQuery dialects.

## Quick Start

1. Load sample data into PostgreSQL (see main [README](../README.md))
2. Open any query file from `postgres/` or `bigquery/`
3. Replace the sample `patient_id` with your own (for patient-scoped queries)
4. Run in your SQL client

## Dialect Differences

Queries are provided in two dialects:
- **`postgres/`** -- Uses double-quoted identifiers, `::TIMESTAMPTZ` casting, `string_to_array` + `unnest`
- **`bigquery/`** -- Uses backtick identifiers, `SAFE_CAST`/`PARSE_TIMESTAMP`, `SPLIT` + `UNNEST`

Business logic is identical across dialects. Only SQL syntax differs.

| `clinical/medication_timeline.sql` | CLIN-03 | Medications with start/end dates, dosage, and status | Patient |
| `clinical/lab_results.sql` | CLIN-04 | Lab values trended by date with interpretation column | Patient |
| `clinical/vital_sign_trends.sql` | CLIN-05 | Blood pressure, heart rate, respiratory rate, O2 sat over time | Patient |

## Sample Data Notes

Queries are designed to work with the included sample data (`sample-data/flat_data.json`). Key characteristics:

- **1 patient** (Elvira Valadez): `6f3bc061-8515-41b9-bc26-75fc55f53284`
- **111 lab results** from a single timestamp, no interpretation values populated
- **116 vital sign observations** including lab-like observations; queries filter to true vitals via LOINC codes
- **5 encounters**, 1 with no start/end time
- **All columns are TEXT/STRING** -- queries CAST to appropriate types for comparisons and math
- **Vital sign timestamps** have a leading comma prefix that queries strip before parsing

## Known Limitations

- `lab_interpretation` is NULL for all records in sample data. The lab results query includes the column but cannot flag abnormals without source-provided interpretation codes.
- `encounter_reference_id` on procedures is NULL for all records. The procedures-by-encounter query uses a direct FK join that will work when populated; a temporal join alternative is provided in comments.
- Labs have no encounter foreign key. The labs-by-encounter query uses timestamp overlap to associate labs with encounters.
- Vital signs table contains lab-like observations (CBC, urinalysis panels). The vital sign trends query filters to true vitals using LOINC codes.
```
