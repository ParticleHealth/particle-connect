**Create `src/observatory/quality.py`:**

This module analyzes the loaded data (the Python dicts, NOT by querying the database) and produces a formatted report. It operates on the data that was just loaded, so it does not need a database connection.

**analyze_quality(data: dict[str, list[dict]], schemas: list[ResourceSchema]) -> list[dict]:**
- For each non-empty schema (schema.is_empty is False):
  - Get records from `data[schema.resource_type]`
  - If no records, skip
  - Calculate:
    - `table_name`: schema.table_name
    - `record_count`: len(records)
    - `column_count`: len(schema.columns)
    - `null_pct`: percentage of NULL/None values across all fields in all records. Formula: (total None values) / (total fields = record_count * column_count) * 100. Use `record.get(col)` to handle missing keys as None.
    - `date_range`: scan all column values that look like dates/timestamps. A value looks like a date if it matches a simple pattern: starts with 4 digits (YYYY). Find min and max of these values (string comparison works since ISO 8601 sorts lexicographically). Format as "{min_date} to {max_date}" or "n/a" if no date-like values found. Truncate to date portion only (first 10 chars: YYYY-MM-DD).
    - `empty_columns`: list of column names where ALL values are None across all records. Report count, not the list.
  - Append result dict to list
- Sort results by record_count descending (largest tables first)
- Return the list of result dicts

**print_quality_report(results: list[dict]) -> None:**
- Use `rich.console.Console` and `rich.table.Table`
- Create table with title "Data Quality Report"
- Columns: "Table" (cyan), "Records" (right-aligned, green), "Columns" (right-aligned), "Null %" (right-aligned, yellow if >50%, red if >80%), "Date Range" (dim), "Empty Cols" (right-aligned, red if >0)
- Add row for each result
- Add a footer row with totals: total records, average null %, total empty columns
- Print to console
- Also print a one-line summary below the table: "Loaded {N} records across {M} tables"

**Update `src/observatory/cli.py`:**
- After `load_all()` completes successfully, call:
  ```python
  from observatory.quality import analyze_quality, print_quality_report
  results = analyze_quality(data, schemas)
  print_quality_report(results)
  ```
- This goes AFTER the load confirmation message but BEFORE the final "Done" message
- The quality report analyzes the Python data dict (not the database), so it works even if the database write had issues with some tables

  - Empty schemas are skipped
- Use real ResourceSchema dataclass instances
- No database needed, no mocking needed -- pure data processing tests

# Particle Flat Data Observatory

Load Particle Health flat data into a local PostgreSQL database and explore it with SQL.

**Quick Start (the 5-minute path):**

Section titled "Quick Start" with numbered steps:

1. **Prerequisites:** Docker, Python 3.11+, pip
2. **Clone and install:**
   ```bash
   cd particle-flat-observatory
   pip install -e .
   ```
3. **Start PostgreSQL:**
   ```bash
   docker compose up -d
   ```
   Note: Tables are created automatically on first startup from `ddl/postgres/create_all.sql`.
4. **Load sample data:**
   ```bash
   particle-pipeline load
   ```
   This loads the included sample data (`sample-data/flat_data.json`) into PostgreSQL. You will see a data quality report showing record counts per table.
5. **Query your data:**
   ```bash
   docker compose exec postgres psql -U observatory -d observatory
   ```
   Then run:
   ```sql
   SELECT given_name, family_name, date_of_birth, gender FROM patients;
   SELECT lab_name, lab_value, lab_unit, lab_timestamp FROM labs LIMIT 10;
   SELECT condition_name, condition_clinical_status FROM problems;
   ```

**Configuration section:**
- Table of environment variables: FLAT_DATA_PATH, PG_HOST, PG_PORT, PG_USER, PG_PASSWORD, PG_DATABASE, LOG_LEVEL
- Note: Copy `.env.example` to `.env` to customize. CLI flags override env vars.
- Note: PG_PORT defaults to 5432. Set to another value if you have a local PostgreSQL running.

**Loading your own data section:**
- Replace `sample-data/flat_data.json` with your Particle GET Flat response
- Or use `--data-path /path/to/your/data.json`
- Data is loaded idempotently: re-running the command replaces existing records per patient

**CLI Reference section:**
- `particle-pipeline load` -- Load flat data into PostgreSQL
- `particle-pipeline load --help` -- Show all options
- `particle-pipeline load --source file --target postgres` -- Explicit mode (default)
- `particle-pipeline load --data-path /path/to/data.json` -- Custom data file
- `particle-pipeline load --verbose` -- Enable debug logging

**Resetting the database section:**
- `docker compose down -v && docker compose up -d` -- Destroy data volume and recreate tables
- Explain why: PostgreSQL init scripts only run on empty data dir
- When to reset: after DDL changes or to start fresh

**Project structure section:**
